# OKR 项目线上部署方案总览

更新时间：2026-06-21

## 1. 结论先行

当前项目是一个标准的 Next.js 全栈应用，依赖 PostgreSQL、Prisma 和飞书 OAuth 回调。基于这些约束，线上部署方案可以分成四类：

1. `本机 + Cloudflare Tunnel`
2. `托管 Node 平台 + 托管数据库`
3. `外部 Linux 主机 + PostgreSQL`
4. `Cloudflare Workers / Pages 重构路线`

其中，`本机 + Tunnel` 是最快、最便宜的临时上线方式，但不能保证本机离线后仍可用。  
`托管 Node 平台 + 托管数据库` 是最快的长期在线方案。  
`外部 Linux 主机 + PostgreSQL` 是最贴近当前仓库设计的稳定方案。  
`Cloudflare Workers / Pages` 适合进一步重构后的低运维路线，不适合当前直接切换。

当前仓库已选定的收口方案是 `Render + Supabase`，执行细节见 [docs/deployment-render-supabase.md](/Users/zhangjiafeng/同步空间/98-AI/codex/OKR/docs/deployment-render-supabase.md)。

## 2. 方案一：本机 + Cloudflare Tunnel

### 方案简介

把当前开发机当作宿主机，Next.js 应用继续在本机运行，通过 Cloudflare Tunnel 暴露一个公网域名。外部用户访问这个公网地址时，流量经由 Cloudflare 转发到你本机。

### 优势

- 零成本
- 改造量最低
- 上线速度最快
- 适合演示、试用、临时验收
- 不需要额外购买云主机

### 劣势

- 本机离线后服务就不可用
- 电脑休眠、断网、重启都会影响可用性
- 不适合长期稳定提供给他人使用
- 本机要承受应用、数据库和外网入口的全部压力

### 适用场景

- 只需要短期演示
- 需要尽快给别人看效果
- 还没决定长期托管平台

### 执行细节

1. 本机继续运行当前 OKR 项目。
2. 确保应用监听固定端口，例如 `3000`。
3. 使用 Cloudflare Tunnel 把本机端口映射到公网域名。
4. 将飞书 OAuth 回调地址改为 Tunnel 分配的公网地址。
5. 做一次完整登录验证、设置页验证、导出验证。

### 风险提示

- 这不是生产级部署。
- 只要本机不在线，别人就无法使用。
- 对于需要稳定在线的 OKR 系统，这个方案只能作为过渡。

## 3. 方案二：托管 Node 平台 + 托管数据库

### 方案简介

把 Next.js 应用部署到支持 Node 运行时的托管平台，再把 PostgreSQL 放到托管数据库服务。常见形态是：

- 应用：Railway / Render / Fly.io 一类平台
- 数据库：平台自带 PostgreSQL 或独立托管 PostgreSQL

### 优势

- 本机可以完全离线
- 成本通常低于自建服务器
- 上线速度快
- 保留现有 Node/Prisma/PostgreSQL 架构，改造量小
- 适合当前项目，不需要重写运行模型

### 劣势

- 不是绝对零成本
- 平台会有配额、冷启动、资源限制
- 运维控制粒度不如自建 Linux 主机
- 长期成本可能高于一次性买主机

### 适用场景

- 你要尽快让别人稳定访问
- 你希望本机不在线也能用
- 你不想重构 Prisma 和数据库访问方式

### 执行细节

1. 选择一个支持 Node.js 和环境变量的托管平台。
2. 部署代码仓库。
3. 配置生产环境变量：
   - `DATABASE_URL`
   - `JWT_SECRET`
   - `NEXT_PUBLIC_APP_URL`
   - `APP_BASE_URL`
   - `ENABLE_MOCK_LOGIN=false`
   - 飞书正式配置项
4. 绑定一个稳定域名。
5. 将飞书 OAuth 回调地址改为该域名下的 `/api/v1/auth/feishu/callback`。
6. 配置数据库迁移和部署后检查。
7. 做 `/api/health`、登录页、导出页的完整验收。

### 风险提示

- 平台的免费额度通常不适合长期生产使用。
- 如果平台对长连接、后台任务、文件系统有约束，要提前确认。

## 4. 方案三：外部 Linux 主机 + PostgreSQL

### 方案简介

使用一台持续在线的 Linux 主机部署 Next.js 应用，数据库使用独立 PostgreSQL，前面配 Nginx 和 HTTPS。这个方案已经是当前仓库文档里固定的首个可交付部署形态。

### 优势

- 最符合当前仓库现有设计
- 对 Prisma、PostgreSQL、飞书 OAuth 的兼容性最好
- 运维路径明确
- 稳定性高
- 最适合后续长期使用

### 劣势

- 需要一台持续在线的主机
- 不是 0 成本
- 需要处理系统运维、证书、备份、监控

### 适用场景

- 你要正式交付给团队使用
- 你要长期稳定运行
- 你接受低成本月费

### 执行细节

1. 准备 Linux 主机、PostgreSQL、Nginx、systemd。
2. 将仓库代码放到固定目录。
3. 配置生产环境变量文件。
4. 执行 `npm run deploy:check`。
5. 执行数据库迁移。
6. 通过 `npm run start:prod` 启动服务。
7. 用 Nginx 反代到内网端口。
8. 配置 HTTPS 证书。
9. 更新飞书 OAuth 回调域名。
10. 做部署后验收。

### 风险提示

- 需要额外资源。
- 如果没有现成主机，就需要采购或申请。

## 5. 方案四：Cloudflare Workers / Pages 重构路线

### 方案简介

把项目重构到更适合 Cloudflare 的运行形态，再部署到 Workers 或 Pages 生态里。这个方向的目标是尽量降低基础设施成本，但它不是当前项目的直接落地方案。

### 优势

- 理论上可以把基础设施成本压得很低
- 适合边缘部署和全球访问
- 如果架构重构完成，后续运维压力会小

### 劣势

- 改造量最大
- 你当前的 Prisma + PostgreSQL + `new PrismaClient()` 写法不是直接可迁移状态
- 飞书 OAuth、数据库访问、后台任务都要重新校准
- 不是最快上线方案

### 适用场景

- 你愿意把它作为重构项目，而不是部署项目
- 你后续想把系统改成更边缘化、更低运维的形态

### 执行细节

1. 调研 Next.js 在 Cloudflare Workers 的支持路径。
2. 改造数据库层，切换到兼容 Workers 的方案。
3. 调整 API Route、OAuth 回调和环境变量。
4. 重新验证飞书集成。
5. 再做完整部署。

### 风险提示

- 这条路不适合当前阶段“快速上线”目标。
- 迁移成本高于直接部署。

## 6. 方案对比

| 方案 | 本机离线后可用 | 成本 | 改造量 | 上线速度 | 稳定性 |
|---|---:|---:|---:|---:|---:|
| 本机 + Cloudflare Tunnel | 否 | 0 | 低 | 最快 | 低 |
| 托管 Node 平台 + 托管数据库 | 是 | 低到中 | 低 | 快 | 中到高 |
| 外部 Linux 主机 + PostgreSQL | 是 | 低到中 | 低 | 中 | 高 |
| Cloudflare Workers / Pages 重构 | 是 | 可低 | 高 | 慢 | 中到高 |

## 7. 我的推荐顺序

如果你现在的目标是“尽快上线并且本机不在线也能用”，推荐顺序如下：

1. `托管 Node 平台 + 托管数据库`
2. `外部 Linux 主机 + PostgreSQL`
3. `本机 + Cloudflare Tunnel` 仅作临时演示
4. `Cloudflare Workers / Pages` 作为后续重构

如果你的目标是“0 成本优先”，那只能接受以下两种折中：

1. `本机 + Cloudflare Tunnel`，但离线即不可用
2. `Cloudflare Workers / Pages`，但需要重构

## 8. 当前项目的现实判断

基于当前仓库状态，这个项目最现实的线上部署策略不是“纯 0 成本一键上线”，而是：

- 短期试用：本机 + Tunnel
- 长期可用：托管 Node 平台或 Linux 主机
- 长期低运维重构：Cloudflare Workers 路线

如果你要的是“现在就可发给别人用，且你本机可以关机”，那就不建议再把精力放在 Tunnel 方案上。
建议直接选一个外部持续在线的平台。
