## 1. 手册适用范围（v1.0）

本手册适用于当前 OKR 团队工具 Web 应用。工具面向公司内部 10-20 人团队，用于围绕《OKR工作法》的每周执行节奏管理季度目标。

当前系统支持：

- 模拟飞书登录
- Super Admin、部门管理者、成员三类角色视角
- 四象限周报看板
- OKR 树状视图
- 周一承诺与周五庆祝入口
- 健康指标查看与健康度更新（v1.1）
- 季度状态查看
- 飞书集成模拟入口
- 季度 Review 查看入口

当前系统暂不支持真实生产数据持久化。页面使用内置演示数据，刷新服务或重启应用后，运行时提交的数据不会作为正式数据库数据保存。

## 2. 启动与访问（v1.0）

### 2.1 启动应用

在项目目录运行：

```bash
npm run build
npm run start
```

启动成功后，终端会显示：

```text
Local: http://localhost:3000
Ready
```

### 2.2 访问地址

在浏览器打开：

```text
http://localhost:3000/login
```

如果页面显示“无法访问此站点”或“无法加载 localhost”，通常表示本地服务没有启动。重新运行 `npm run start` 后刷新页面。

### 2.3 开发模式说明

当前机器上使用 `npm run dev` 可能出现 `EMFILE: too many open files, watch` 文件监听警告。建议优先使用生产预览方式：

```bash
npm run build
npm run start
```

## 3. 登录与角色切换（v1.0）

### 3.1 登录方式

进入登录页后，选择一个模拟角色：

- 超级管理员
- 部门管理者
- 成员

点击角色卡片后，系统会模拟飞书授权并进入 `/dashboard`。

### 3.2 角色说明

| 角色 | 适合验证的功能 |
|---|---|
| 超级管理员 | 全公司视图、季度、设置、飞书模拟集成 |
| 部门管理者 | 部门 OKR、健康指标、部门成员视角 |
| 成员 | 个人 OKR、周仪式、个人看板 |

### 3.3 退出登录

进入系统后，页面右上角点击“退出”。退出后应回到登录状态。

## 4. 四象限周报看板（v1.0）

路径：

```text
/dashboard
```

四象限看板是每周例会的核心页面，分为四个区域。

### 4.1 左上：本周承诺

展示当前用户本周 Top 3 承诺事项。

如果尚未提交，会显示“本周承诺尚未提交”和快速入口。

### 4.2 右上：本季度 OKR 与信心值

展示公司级、部门级、个人级 OKR 树。

每个 KR 旁边会显示信心值状态：

| 状态 | 含义 |
|---|---|
| 绿色 | 信心值 7-10 |
| 黄色 | 信心值 4-6 |
| 红色 | 信心值 1-3 |
| 灰色 | 本周未更新 |

### 4.3 左下：当周团队心情

展示当前用户本周五庆祝填写的心情和完成事项。

如果尚未提交，会显示“本周心情尚未提交”和快速入口。

### 4.4 右下：健康指标（v1.1）

展示公司或部门健康指标的最新状态。

| 状态 | 含义 |
|---|---|
| healthy | 健康度 `7.0-10.0` |
| warning | 健康度 `4.0-6.9` |
| exceeded | 健康度 `0-3.9` |

看板中的健康指标使用统一 `0-10` 健康度评分，不再要求用户理解每个指标不同单位的阈值。

## 6. 周仪式入口（v1.0）

路径：

```text
/weekly
```

周仪式页面包含周一承诺和周五庆祝两个区域。

### 6.1 周一承诺

用于填写本周 Top 3 优先级。

建议填写标准：

- 每条优先级不超过 100 字
- 必须是本周真实要推进的事项
- 应与 OKR 有关，但不必等同于 KR

### 6.2 周五庆祝

用于填写本周完成事项、障碍和心情。

心情选项：

- 充满能量
- 稳步前进
- 平静
- 有些疲惫
- 需要支持

当前版本的页面表单主要用于演示流程入口，真实提交后的完整持久化能力会在接入正式数据库后完善。

## 7. 健康指标（v1.1）

路径：

```text
/health
```

健康指标用于声明团队不愿为了完成 OKR 而牺牲的业务底线。

页面展示：

- 指标名称
- 指标说明
- 公司级或部门级
- 负责人
- 健康度类型
- 当前健康度
- 最新状态

### 7.1 健康度评分规则（v1.1）

当前版本统一使用 `0-10` 健康度评分：

| 分数范围 | 状态 | 含义 |
|---|---|
| `7.0-10.0` | healthy | 健康 |
| `4.0-6.9` | warning | 预警 |
| `0-3.9` | exceeded | 超限 |

录入规则：

- 最小值为 `0`。
- 最大值为 `10`。
- 最多保留小数点后 1 位。
- 如果输入了更多小数位，系统会按 1 位小数保存，例如 `7.26` 会保存为 `7.3`。

### 7.2 更新健康指标（v1.1）

有权限的用户可在健康指标卡片中填写“最新数值”并提交。

可更新人员：

- 指标负责人
- 超级管理员

无权限用户只能查看指标，不能更新。

提交后系统会自动计算状态：

- `7.0-10.0` 自动显示为 `healthy`
- `4.0-6.9` 自动显示为 `warning`
- `0-3.9` 自动显示为 `exceeded`

### 7.3 与旧版阈值说明的关系（v1.1）

v1.0 手册中的 `gte / lte / between` 阈值类型不再作为当前页面主要操作口径。

当前页面面向用户统一展示为 `0-10 健康度`。旧阈值类型仅作为底层技术兼容能力保留，不建议在日常操作中按旧单位理解健康指标。

健康指标与 KR 分开管理，不应把健康指标当成 KR 使用。

## 12. 常见问题（v1.0）

### 12.1 页面打不开，提示无法加载 localhost

原因通常是本地服务没有启动。

处理方式：

```bash
npm run build
npm run start
```

然后刷新：

```text
http://localhost:3000/login
```

### 12.2 点击登录后没有进入系统

先确认服务仍在运行，再刷新登录页重新选择角色。

如果仍无法进入，检查登录接口是否可用：

```bash
curl -i -X POST http://localhost:3000/api/v1/auth/mock-login \
  -H 'Content-Type: application/json' \
  -d '{"role":"super_admin"}'
```

返回 `200 OK` 表示登录接口正常。

### 12.3 为什么不是直接飞书登录

当前阶段按开发计划先使用模拟飞书登录，目的是先验证 OKR 产品流程。真实飞书 OAuth 和通讯录同步会在后续集成阶段替换。

### 12.4 数据会保存吗

当前页面使用演示数据和内存数据。服务重启后，运行时新增或修改的数据不会作为正式数据保留。

### 12.5 为什么建议使用 npm run start 而不是 npm run dev

当前机器使用开发模式时可能触发文件监听上限警告。生产预览模式更稳定：

```bash
npm run build
npm run start
```
